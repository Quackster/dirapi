============================================================
1. OVERALL ARCHITECTURE
============================================================

This is NOT a Director Xtra DLL. This is the CORE Director runtime engine
(dirapi.dll). It implements the RIFX/DIR file format parser, the Lingo
scripting engine, the JavaScript engine, the property dispatch system,
the Xtra loading infrastructure, and the cast member management system.

There is NO single "Lingo method dispatch table" with string comparisons
like "setPixels", "getPixels", "applyFilter", etc. Those methods belong
to individual Xtras (separate DLLs), not to dirapi.dll itself.

Instead, dirapi.dll uses a NUMERIC PROPERTY ID DISPATCH system. Properties
and methods are identified by integer IDs (not strings), and dispatch is
done via large switch statements on these numeric IDs.

3. FOURCC (4-CHARACTER CODE) CONSTANTS
============================================================

The following FourCC codes are used throughout for Director file format:

  Container types:
    0x52494658 = "RIFX"  - Big-endian Director file container
    0x4647444D = "FGDM"  - Director movie container (alternative)
    0x46474443 = "FGDC"  - Director cast library container (alternative)
    0x4D563933 = "MV93"  - Movie Version 93 (standard Director movie format)

  Chunk types:
    0x696D6170 = "imap"  - Initial memory map (first chunk in RIFX)
    0x6D6D6170 = "mmap"  - Memory map (chunk index)
    0x6A756E6B = "junk"  - Padding/junk chunk
    0x66726565 = "free"  - Free/deleted chunk
    0x4B45592A = "KEY*"  - Key table (maps chunk resources)
    0x43415374 = "CASt"  - Cast member data
    0x4C6E616D = "Lnam"  - Lingo name table
    0x4C736372 = "Lscr"  - Lingo script data
    0x4C637478 = "Lctx"  - Lingo context
    0x736E6420 = "snd "  - Sound data chunk

Key dispatch function for chunk type handling:
  FUN_680031E0 (line 2950, addr 0x680031E0):
    Maps FourCC chunk types to handler vtable pointers:
      "RIFX" -> PTR_FUN_68188084 (RIFX container handler)

      "junk" -> PTR_FUN_6818805C (junk chunk handler)

4. PROPERTY DISPATCH SYSTEM (NUMERIC ID-BASED)
============================================================

Director does NOT dispatch Lingo properties/methods by string name at
runtime. Instead, it uses a two-phase system:

Phase 1 - String to ID Resolution:
  FUN_68031CA0 (line 49954, addr 0x68031CA0):
    Core symbol table lookup. Takes a string and its length, hashes it
    via FUN_68039820, then looks up in FUN_680533F0. Returns an integer
    property ID.

  FUN_68038D10 (line 56966, addr 0x68038D10):
    Convenience wrapper around FUN_68031CA0. Calculates strlen first,
    then calls FUN_68031CA0. Used for property name -> ID conversion.
    Examples seen:
      FUN_68038d10(param_1, "width")
      FUN_68038d10(param_1, "height")
      FUN_68038d10(param_1, "directToStage")
      FUN_68038d10(param_1, "antiAliasThreshold")
      FUN_68038d10(param_1, "is3dCastMember")
      FUN_68038d10(param_1, "Compare")

Phase 2 - Numeric ID Dispatch:
  Hundreds of switch statements dispatch on numeric property IDs.
  The param_3 or param_4 parameter typically carries the property ID.
  Common property ID ranges observed:

    0x39-0x79: Boolean properties (e.g., visibility, editability)
    0x48-0x58: Position/dimension properties (locH, locV, width, height)
    0xBC: Rectangle property (rect)
    0x99: Point property
    0x139: Byte-sized properties
    0x168-0x16C: Flag properties
    0x1A3-0x1BC: Capability properties
    0x1AF-0x1B6: Size/format properties
    0x1E2-0x1F5: Point properties
    0x338: Named properties
    0x35B: Rectangle with set support
    0x3D2-0x3D4: Float properties

  Major property dispatch functions:
    FUN_6800D7B0 (line 14548): Switch on param_3, handles properties
        0x1A3, 0x1A4, 0x1A6, 0x1A7, 0x1A9, 0x1BC.
    FUN_6800DA30 (line 14670): Large dispatch with nested if-else and
        switch on param_3, handles 0x39-0x35B range.
    FUN_6800D990 (line 14635): Handles float properties 0x3D2-0x3D4.

  The pattern in each case is:
    if (param_2 == 0) -> GET property (read into param_4 output buffer)
    if (param_2 != 0) -> SET property (parse param_4, write to object)

============================================================
SUMMARY
============================================================

dirapi.dll is the core Director runtime, not an Xtra. It does not contain
the specific Lingo method strings like "setPixels", "getPixels",
"applyFilter", "perlinNoise", "openURL", or "tapGesture" -- those would
be in individual Xtra DLLs (e.g., an imaging Xtra, a network Xtra, etc.).

The DLL's dispatch architecture is:
  1. RIFX file I/O via exported ordinals 10-23
  2. Chunk type routing via FourCC codes and FUN_680031E0
  3. Property access via numeric ID switch statements (not string dispatch)
  4. Xtra communication via COM-like vtable calls with string property names
  5. JavaScript atom registration via FUN_68138710
  6. Lingo object type registration via FUN_6811E190

The numeric property IDs are resolved from strings at initialization time
by FUN_68031CA0/FUN_68038D10 (symbol table hash lookup), then all runtime
dispatch uses the integer IDs in switch statements for performance.
